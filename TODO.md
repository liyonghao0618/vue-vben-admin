# 守护桑榆 TODO

## 文档说明

本 TODO 基于两部分整理：

- `需求文档.md` 中的 V1 需求范围
- 当前项目代码中的实际完成情况（前端页面、接口封装、FastAPI 业务 API、数据库模型）

目标不是重新写一份需求文档，而是把项目当前阶段真正需要推进的事项列清楚，方便比赛开发按优先级往前做。

## 当前进度判断

### 已有基础

- 前端已经完成四类角色的路由拆分：老年端、子女端、社区端、管理端
- 前端已经落地大部分业务页面骨架，并且接入了对应 API 封装
- 后端已完成认证、角色权限、绑定关系、风险识别、风险告警、通知、社区工单、管理端配置等核心 API
- 后端已有数据库模型、Alembic 迁移、演示账号、规则版风险识别闭环
- 短信/通话风险识别已经能输出风险等级、命中规则、原因说明、处置建议，并自动触发告警/通知/工单

### 当前主要短板

- 前端不少页面仍偏“展示型”或“联调型”，离比赛演示中的完整业务闭环还差最后一层交互打磨
- 部分统计、趋势、老人名称、宣教计划等仍有明显演示数据/前端拼装痕迹
- 老年端需求里强调的“适老化体验、大字、高对比、语音播报”还没有形成很完整的产品呈现
- 通话识别、短信识别目前是规则版文本识别，更适合比赛 V1 演示，但离真实采集和智能分析还有差距
- 管理后台虽然接口和页面都已铺开，但“可配置、可运营、可追踪”的完整感还需要补足

## 第一优先级

这些建议优先完成，最直接影响比赛演示效果。

### 1. 打通前端四端核心闭环

- [ ] 老年端：完成“风险提醒 -> 一键求助 -> 通知子女/社区 -> 记录结果”的可视化闭环演示
- [ ] 子女端：完成“接收高风险通知 -> 查看告警详情 -> 发送远程提醒 -> 已读/跟进”的闭环操作
- [ ] 社区端：完成“查看重点老人 -> 查看工单 -> 更新处理状态/备注 -> 完成处置”的闭环操作
- [ ] 管理端：完成“新增/编辑风险规则、内容、系统配置”的页面交互闭环

### 2. 清理页面中的演示占位数据

- [ ] 把首页卡片、趋势图、统计数字中能改为真实接口返回的部分尽量改成真实数据
- [ ] 减少页面里的硬编码老人姓名、社区名称、固定提示语
- [ ] 检查各页面“空状态 / 加载状态 / 接口失败提示”，避免比赛演示时出现空白页

### 3. 强化适老化表现

- [ ] 老年端统一大字号、大按钮、高对比度视觉规范
- [ ] 把“适老设置”真正影响到页面展示，而不只是设置表单
- [ ] 至少补一个可演示的语音播报/朗读方案
- [ ] 优化老年端首页与风险提醒页，让用户一眼看懂“是否危险、下一步做什么”

### 4. 准备稳定可演示的数据链路

- [ ] 检查演示账号的角色、菜单、首页跳转是否都正确
- [ ] 准备一组稳定的短信诈骗样例和通话诈骗话术样例，确保每次演示都能稳定触发中高风险
- [ ] 确保高风险后一定能看到子女通知和社区工单生成结果
