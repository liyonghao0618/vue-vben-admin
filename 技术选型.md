# 守护桑榆——反电诈 AI 助手
## 技术选型

**版本**：V1.0  
**文档目标**：结合当前需求文档与现有仓库基础，明确项目的实际技术路线、分层职责与阶段性实现方案，避免继续停留在泛选型阶段。

## 1. 选型结论

本项目当前已经具备 `vue-vben-admin` 前端工程基础，因此技术选型不再建议切换到其他前后端一体化脚手架，而应采用“前后端分离 + 风险识别服务独立演进”的方案。

**最终建议如下：**

- 前端管理与业务页面：`vue-vben-admin`（Vue 3 + TypeScript）
- 后端业务服务：`FastAPI`
- 数据库：`PostgreSQL`
- 缓存与消息能力：`Redis`
- 实时通知：`WebSocket`
- V1 风险识别：规则引擎 + 关键词匹配 + 黑白名单
- V2 风险识别增强：`Whisper` + `Transformers` / 其他可替换模型服务

该方案的核心原则是：

1. 先把“提醒、通知、协同、后台配置”业务闭环跑通。
2. 先做可上线、可演示、可迭代的 V1.0，而不是一开始投入过多精力在模型训练与复杂语音能力上。
3. 将 AI 风险识别设计为独立模块，避免后期模型迭代影响主业务系统。

## 2. 为什么这样选

### 2.1 前端不再更换基座

当前仓库已经是 `vue-vben-admin` 工程，说明前端脚手架、权限菜单、布局体系、工程规范已经具备基础能力。继续改为 `django-vue3-admin` 或其他整套后台方案，会带来以下问题：

- 与现有仓库方向冲突
- 增加重构成本
- 推迟业务功能开发
- 让技术选型重新回到不确定状态

因此，前端基座直接沿用当前仓库是最合理的路径。

### 2.2 后端选择 FastAPI 而非 Django 全家桶

本项目核心难点不在传统后台 CRUD，而在以下几类能力：

- 多角色业务接口
- 风险识别结果接入
- 通知推送
- 后续模型服务扩展
- 实时事件处理

`FastAPI` 更适合承担这种“业务接口 + 异步任务 + 模型服务集成”的职责，优势包括：

- 与 Python 风险识别生态集成自然
- 接入 AI / NLP / 语音识别更直接
- 编写 REST API 成本较低
- 对异步通知、任务处理、WebSocket 支持更友好

因此，后端建议采用 `FastAPI`，而不是再引入另一个完整后台框架。

## 3. 系统总体架构

建议采用如下分层：

### 3.1 前端层

- 老年端页面
- 子女端页面
- 社区端页面
- 管理端页面

统一基于 `vue-vben-admin` 承载，按角色做菜单、路由和权限区分。

### 3.2 业务后端层

基于 `FastAPI` 提供：

- 登录认证
- 用户与角色管理
- 亲属绑定
- 风险告警管理
- 通知记录管理
- 工单与社区协同
- 内容与知识库管理
- 风险规则配置

### 3.3 风险识别层

独立为单独模块或服务，负责：

- 短信文本分析
- 通话文本分析
- 风险等级计算
- 命中原因输出
- 建议动作生成

V1 采用规则方式实现，V2 再逐步接入模型能力。

### 3.4 基础设施层

- `PostgreSQL`：存储业务主数据
- `Redis`：缓存、会话、通知队列、热点数据
- `WebSocket`：高风险事件实时推送

## 4. 各模块技术选型

### 4.1 前端

**选型**：`vue-vben-admin`

**原因**：

- 当前仓库已具备现成工程基础
- 适合快速搭建后台、列表、表单、统计页面
- 权限、菜单、路由、主题能力较成熟
- 便于快速支撑社区端和管理端页面建设

**建议落地方式**：

- 先优先完成社区端、子女端、管理端
- 老年端可继续沿用同一前端工程，但在交互和视觉上单独做适老化页面
- 老年端页面需强化大字号、大按钮、高对比度和少步骤交互

### 4.2 后端 API

**选型**：`FastAPI`

**原因**：

- 与 Python AI 生态兼容度高
- 编写接口与数据校验效率高
- 更适合后续拆分风险识别服务
- 支持异步接口与 WebSocket

**建议配套**：

- ORM：`SQLAlchemy 2.x`
- 数据校验：`Pydantic`
- 数据库迁移：`Alembic`
- 认证：JWT

### 4.3 数据库

**选型**：`PostgreSQL`

**原因**：

- 适合复杂业务关系建模
- 便于做统计、筛选、报表
- 支持较好的 JSON 扩展字段能力
- 适合后续审计、记录追踪和多维查询

**核心表建议**：

- 用户表
- 角色表
- 用户角色关联表
- 老人与亲属绑定表
- 风险告警表
- 短信识别记录表
- 通话识别记录表
- 通知记录表
- 工单表
- 社区处置记录表
- 风险规则表
- 宣教内容表

### 4.4 缓存与消息

**选型**：`Redis`

**用途**：

- 登录态缓存
- 热点配置缓存
- 告警推送缓冲
- 通知队列
- 临时验证码或一次性状态

如果后续告警量增大，可进一步引入更明确的消息队列，但 V1 阶段 `Redis` 已足够支撑。

### 4.5 实时通知

**选型**：`WebSocket + Redis`

**适用场景**：

- 高风险告警推送子女端
- 高风险事件同步社区端
- 后台实时刷新告警状态

**说明**：

V1 可先实现站内实时通知与告警列表刷新；短信、电话外呼、第三方消息等能力可作为后续增强项。

### 4.6 风险识别

#### V1.0 方案

**选型**：规则引擎 + 关键词匹配 + 风险词库

**覆盖内容**：

- 冒充公检法
- 验证码套取
- 链接诱导点击
- 转账汇款诱导
- 中奖返利
- 熟人冒充

**原因**：

- 实现成本低
- 可解释性强
- 便于在后台配置和迭代
- 更适合先做验收与演示版本

#### V1.1 / V2.0 增强方案

**候选技术**：

- 语音转写：`Whisper`
- 文本分类：`Transformers` + 中文预训练模型

**适用方向**：

- 通话内容转写后风险识别
- 更复杂的短信语义分类
- 风险原因解释增强
- 模型效果持续迭代

**结论**：

模型能力应作为增强模块逐步接入，不建议阻塞 V1.0 主线开发。

### 4.7 工单与协同

**选型**：自研业务模块

虽然存在一些开源工单系统，但当前项目的工单流程与“风险告警、老人信息、社区处置、宣教联动”高度耦合，直接复用独立工单系统通常会带来以下问题：

- 用户体系重复
- 权限体系重复
- 数据模型难以统一
- 前后端交互不够贴合本项目场景

因此，V1 建议直接在现有业务系统内实现轻量工单模块，包括：

- 创建工单
- 分配处理人
- 记录跟进
- 上传附件
- 完成归档

### 4.8 内容与知识库

**选型**：后台内容管理 + 数据库存储

**内容范围**：

- 诈骗案例
- 防骗提示模板
- 风险词库
- 社区宣教素材

**原因**：

- 需求文档明确要求后台可配置
- 这是 V1 就必须具备的运营能力
- 与风险识别规则和老人端提醒文案高度相关

## 5. V1.0 实施策略

### 5.1 优先做业务闭环

V1.0 建议优先完成以下闭环：

1. 用户登录与角色区分
2. 老人绑定亲属
3. 短信 / 通话风险结果接入
4. 风险分级展示
5. 子女告警通知
6. 社区工单处置
7. 后台规则与内容配置

### 5.2 暂缓重投入事项

以下能力不建议作为 V1.0 的阻塞项：

- 复杂模型训练
- 方言识别优化
- 独立大规模工单平台接入
- 多种外部消息通道打通
- 复杂智能推荐与主动预警

### 5.3 推荐开发顺序

1. 数据模型设计
2. API 设计
3. 管理端与社区端页面
4. 子女端核心页面
5. 老年端适老化页面
6. 风险规则引擎
7. WebSocket 通知
8. 模型能力增强

## 6. 推荐技术栈总览

| 模块 | 最终选型 | 说明 |
|---|---|---|
| 前端工程 | `vue-vben-admin` | 继续沿用当前仓库 |
| 前端框架 | `Vue 3` + `TypeScript` | 当前工程基础 |
| 后端服务 | `FastAPI` | 适合业务接口和 AI 集成 |
| ORM | `SQLAlchemy 2.x` | 配合 FastAPI 使用 |
| 数据校验 | `Pydantic` | 请求与响应模型定义 |
| 数据迁移 | `Alembic` | 数据库版本管理 |
| 数据库 | `PostgreSQL` | 核心业务数据存储 |
| 缓存 / 消息缓冲 | `Redis` | 会话、缓存、推送缓冲 |
| 实时通知 | `WebSocket` | 高风险事件实时推送 |
| V1 风险识别 | 规则引擎 + 风险词库 | 先实现可解释、可配置能力 |
| V2 风险识别 | `Whisper` + `Transformers` | 后续增强语音与语义识别 |

## 7. 最终建议

本项目当前最合理的技术路线不是继续比较各种后台框架，而是基于现有 `vue-vben-admin` 仓库，尽快进入业务落地阶段。

因此，技术选型建议正式收敛为：

- 前端固定为 `vue-vben-admin`
- 后端固定为 `FastAPI`
- 数据层固定为 `PostgreSQL + Redis`
- 风险识别采用“V1 规则优先、V2 模型增强”的分阶段方案
- 工单、规则、内容管理优先自研并与主业务统一

该方案最符合当前项目状态，也最有利于在较短周期内完成一个可演示、可验收、可继续迭代的 V1.0。
