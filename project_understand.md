总的来说，我们的反诈系统不是“实时监听所有通话”，而是一个围绕可疑通信事件展开的补充型守护系统。

系统自身不重点解决“模型如何识别诈骗”这个问题。音频反诈识别能力已经在 `read_audio_guard_improved` 中做好准备，本项目更关注业务侧如何发起识别、接收识别结果，并把结果转化为老人端提醒、家属端通知、社区端工单和后台管理记录。

换句话说，本系统的核心职责是：

* 在合适的时机发起反诈分析
* 将通话、短信、号码、联系人等业务上下文整理成模型可用的输入
* 调用已经准备好的反诈识别能力
* 根据识别结果触发守护业务反应
* 用 Web 端把整条反诈链路清晰、可信地展示出来

核心工作流程是：

1. 来电 / 短信进入手机端

系统先判断来源号码：

* 是否在被守护者通讯录 / 家庭白名单中
* 是否在官方可信号码库中
* 是否属于陌生号码、异常号码、可疑短信来源

安全号码直接放行，不做推理。

2. 陌生通话触发本地规则筛查

如果来电号码不在白名单，也不在官方号码库中，手机端开启保护逻辑：

* 通话开始后自动录音
* 如果通话时间不足 1 分钟，认为诈骗价值较低
* 通话结束后直接删除本地录音
* 不上传云端，减少隐私风险和算力浪费

3. 超过 1 分钟的陌生通话发起分析

如果陌生通话持续超过 1 分钟：

* 通话结束后保存录音
* 后端接收录音和通话上下文
* 系统发起对 `read_audio_guard_improved` 的调用
* 反诈识别模块返回诈骗风险等级、诈骗类型、证据片段和解释

4. 反诈识别模块完成风险判决

`read_audio_guard_improved` 主要负责判断：

* 是否存在诈骗风险
* 属于哪类诈骗，如冒充公检法、刷单返利、客服退款、虚假投资、熟人借钱等
* 通话中有哪些高风险话术
* 给出可供业务系统使用的风险结论和解释依据

本系统不在这一层重复实现模型推理逻辑，而是把识别结果作为业务决策输入。

5. 系统根据判决结果触发守护业务

如果反诈识别模块判定存在风险，业务系统负责做出反应：

* 通知子女端 / 守护人
* 向被守护者端弹出风险提醒
* 后台管理端记录事件
* 必要时由社区工作人员或民警查看风险记录
* 形成从风险发现到协同处置的完整闭环

6. 短信侧逻辑类似

短信进入后先做规则判断：

* 官方号码、白名单短信放行
* 陌生短信、含链接短信、转账/验证码/退款/投资等关键词短信进入风险识别
* 可疑短信由业务系统发起模型分析
* 高风险短信触发提醒和守护通知

一句话概括：

我们的系统是国家反诈 App 的补充防线，重点守护老人群体的陌生通话和可疑短信，通过本地规则减少误报和隐私上传，再调用已经准备好的反诈识别能力完成风险判断，最终由业务系统触发提醒、通知、记录和协同处置。

# 项目总结

它通过 Web 模拟老人手机端的短信和电话输入，通过 FastAPI 后端组织业务流程，调用 `read_audio_guard_improved` 完成真实音频反诈识别，再将结果包装成老人端提醒、家属端通知、社区端工单和管理端记录。

项目的核心不是证明我们能控制手机系统，也不是重复证明模型本身能完成推理，而是证明我们能把“可疑通信内容 -> 发起反诈识别 -> 接收风险结论 -> 守护通知 -> 协同处置”这条业务链路完整跑通，并用清晰、可信、漂亮的方式展示出来。
