# P0 架构结论

## 1. 最终产品形态

P0 结论：先采用“一个应用双角色”的 `uni-app` 方案。

原因：

- 更适合当前 MVP 和比赛演示节奏
- 能在一个工程内快速验证聊天、提醒、求助和 AI 联动闭环
- 后续如需长期产品化，可再按角色拆分为两个独立应用

## 2. 页面组织方案

目录约定：

- `src/pages/index`：启动页 / 角色分流页
- `src/pages/elder`：老年端页面
- `src/pages/guardian`：子女 / 守护人端页面
- `src/components`：跨角色复用组件
- `src/api`：接口模块与请求封装
- `src/store`：跨页面状态
- `src/utils`：导航、配置、通用工具

原则：

- 角色页面按目录拆分，减少条件判断蔓延
- 公共能力统一放到 `components`、`api`、`store`、`utils`
- 先保持扁平结构，P1 再继续按功能域细分

## 3. 状态管理方案

采用 `Pinia`。

理由：

- 与 Vue3 组合式 API 配合自然
- 可同时服务 H5 和后续微信小程序构建目标
- 比自定义全局对象更容易沉淀登录态、角色态、风险记录和会话数据

P0 承载的数据：

- 当前角色
- 当前演示用户信息
- 风险记录 mock 数据

P1 将继续接入：

- 登录态与 token
- 联系人 / 会话 / 通知列表
- 风险详情缓存

## 4. 导航方案

采用 `uni-app` 原生页面导航：

- 启动页通过 `uni.navigateTo` 进入角色首页
- 当前阶段不引入复杂路由库
- 如果后续首页结构稳定，可再评估 `tabBar` 承载主导航

原因：

- 保持 H5 和微信小程序行为一致
- 减少额外依赖和路由兼容问题

## 5. UI 组件方案

P0 方案：原生 `uni-app` 基础组件 + 项目自定义基础组件。

当前做法：

- 使用 `view`、`text`、`button` 等跨端基础组件
- 建立 `src/components/ui/ss-card.vue` 作为项目级基础卡片容器
- 统一配色变量放在 `src/uni.scss`

后续策略：

- P1 如需快速扩表单、列表、弹层，再按需引入 `uni-ui`
- 不在 P0 提前引入过多 UI 依赖，先保证跨端一致和可控样式

## 6. 接口边界

### 主业务系统

主业务系统负责：

- 登录鉴权
- 用户信息
- 老人与守护人的绑定关系
- 风险通知 / 告警记录
- 一键求助 / 处置记录
- 文本聊天消息的业务落库与会话列表

建议移动端命名空间：

- `/api/mobile/auth/*`
- `/api/mobile/users/*`
- `/api/mobile/relations/*`
- `/api/mobile/alerts/*`
- `/api/mobile/sos/*`
- `/api/mobile/chat/*`

### ai-api-service

AI 服务负责：

- 单条文本诈骗识别
- 聊天记录批量分析
- 链接风险识别
- 后续通话摘要分析

当前边界：

- `POST /api/fraud-detect`
- `POST /api/fraud-detect/chat-log`
- `POST /api/fraud-detect/link`

原则：

- 移动端不直接承载复杂识别逻辑
- 移动端提交结构化文本或链接，服务端返回风险等级、原因、建议

## 7. 聊天能力实现方案

P0 结论：P1 先做“业务系统代管的文本聊天 MVP”。

实现顺序：

1. 先完成会话列表和单聊消息页
2. 通过主业务系统 API 拉取 / 发送消息
3. 在消息发送或接收后触发 AI 风险识别
4. 把识别结果回写到消息展示层和风险通知流

暂不在 P0 引入 IM SDK，原因：

- 当前目标是先把守护业务闭环跑通
- 先验证消息结构、风控链路和双角色页面逻辑

## 8. App 内语音通话方案

P0 结论：P1 预留 VoIP SDK 接入位，不在当前阶段直接实现。

建议方向：

- 评估腾讯云 TRTC、声网 Agora 或其他支持小程序 / App 的语音方案
- H5 阶段仅保留入口和状态位，不承诺浏览器端直接完成稳定语音通话

原因：

- 语音通话对平台能力和权限更敏感
- 与 H5 先开发的节奏相比，先完成页面、状态和接口边界更重要

## 9. 开发策略

当前策略：

- 先用 H5 完成页面开发、状态联调和接口调试
- 当整体进度达到约 95% 时，再迁移到微信小程序做专项兼容测试

这样做的原因：

- 开发调试速度更快
- 更方便团队协作和演示
- 小程序阶段集中处理权限、样式和平台差异即可
