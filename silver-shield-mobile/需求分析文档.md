# Silver Shield Mobile Apps 需求分析文档

# 先用H5开发和调试,最终进度达到95%的时候移动到微信小程序进行测试

## 一、项目背景

当前“守护桑榆 / Silver Shield AI”项目已经具备：

- Web 前端多角色业务系统
- 后端业务 API
- 独立 `ai-api-service` 诈骗识别服务

但现阶段系统主要以 Web 端演示和管理为主，缺少真正贴近用户日常通信场景的移动端入口。

而诈骗高发场景往往发生在：

- 手机聊天
- 手机通话
- 链接点击
- 陌生联系人互动

因此需要新增一个独立移动端项目，让老年用户与子女 / 守护人能在手机上进行聊天与通话，并在过程中接入反诈分析能力。

## 二、项目目标

构建一个双用户端移动应用项目，使其具备以下目标：

1. 支持老年用户与子女 / 守护人在手机端通信
2. 支持文本聊天和语音通话
3. 支持将聊天文本、链接、通话摘要提交给 AI 服务分析
4. 在发现风险时及时提醒、求助、通知家属
5. 与主业务系统联动，形成“识别 -> 告警 -> 处置”的闭环

## 三、核心问题

当前需要解决的核心问题不是“做一个通用社交软件”，而是：

- 如何让老人愿意用
- 如何让家属能看见风险
- 如何把通信场景接入诈骗识别
- 如何在风险发生时快速联动处置

因此，这个项目的本质是：

`通信能力 + 反诈识别 + 家庭守护联动`

## 四、目标用户

### 1. 老年用户

特征：

- 对复杂交互不敏感
- 对诈骗内容辨识能力较弱
- 更依赖清晰提示和亲属帮助

核心诉求：

- 操作简单
- 看得清
- 遇到风险时能立即获得帮助

### 2. 子女 / 守护人

特征：

- 希望及时获知老人风险情况
- 希望可以远程提醒或干预
- 希望有可追踪的风险记录

核心诉求：

- 消息及时
- 风险清晰
- 处置方便

## 五、业务边界

### 本项目负责

- 移动端用户登录与角色体验
- 聊天与通话界面
- 消息展示、提醒、求助交互
- 将可分析内容提交到后端 / AI 服务
- 展示风险识别结果与处理建议

### 主业务系统负责

- 用户体系
- 家庭绑定关系
- 风险事件、通知、工单、管理配置
- 社区与后台联动能力

### `ai-api-service` 负责

- 文本诈骗识别
- 聊天记录分析
- 链接风险分析
- 人工复核与审计能力
- Qwen 模型接入

## 六、产品形态分析

这个项目有两种实现思路：

### 方案 A：一个应用双角色

优点：

- 工程成本更低
- 更适合 `uni-app` 快速做 MVP
- 登录后根据角色切换页面即可

缺点：

- 老年端和子女端的体验容易互相妥协
- 适老化设计需要额外注意隔离

### 方案 B：一个仓库两套前端

例如：

- 老年端应用
- 子女端应用

优点：

- 角色清晰
- UI 与交互可完全按角色优化
- 更适合长期产品化

缺点：

- 工程维护成本更高
- 重复开发内容较多

### 建议

如果目标是比赛演示和短期落地，可以优先：

- 先做“一个应用双角色”的 `uni-app` MVP

如果目标是更完整的真实产品形态，则建议：

- 后续再拆分为两个独立应用

## 七、功能需求分析

### 1. 通用能力

- 登录 / 注册 / 鉴权
- 用户资料与角色信息
- 联系人与绑定关系同步
- 通知与风险提醒
- 基础设置

### 2. 老年端功能

- 首页
- 联系人列表
- 聊天会话
- 通话入口
- 风险提醒弹层 / 页面
- 一键求助
- 防骗知识
- 适老化设置

### 3. 子女端功能

- 监护总览
- 老人列表
- 风险通知
- 风险详情
- 与老人聊天
- 与老人通话
- 远程提醒
- 求助记录查看

### 4. AI 识别相关功能

- 单条消息分析
- 会话级聊天分析
- 链接风险分析
- 通话摘要分析
- 高风险提示
- 分析记录留痕

## 八、重点场景分析

### 场景 1：聊天诈骗识别

流程：

1. 老年用户收到聊天消息
2. 消息或会话被送到 AI 服务识别
3. 系统判断存在诈骗风险
4. 老年端提示“存在风险，请勿转账 / 点击链接”
5. 子女端同步收到风险通知

### 场景 2：通话后风险提示

流程：

1. 老人与陌生联系人发生通话
2. 通话摘要或转写文本被提交分析
3. 系统识别为高风险
4. 老年端出现显著提醒
5. 子女端与社区端收到联动通知

### 场景 3：一键求助

流程：

1. 老人识别到可疑情况
2. 点击一键求助
3. 子女端收到提醒
4. 主业务系统生成记录
5. 必要时进入社区工单流转

## 九、非功能需求

### 1. 易用性

- 老年端必须简洁
- 字体和按钮必须足够大
- 风险提示必须显眼

### 2. 稳定性

- 网络差时不能直接崩
- 关键请求要有失败提示
- 风险识别失败时要有兜底提示

### 3. 安全性

- 敏感数据传输加密
- API 鉴权
- 风险数据和隐私数据分级处理

### 4. 合规性

- 明确采集范围
- 明确用户授权
- 谨慎处理系统短信 / 系统电话接入能力

## 十、技术建议

推荐优先使用：

- `uni-app`

原因：

- 更适合快速做双角色移动应用 MVP
- 页面和业务逻辑开发效率高
- 对比赛演示、原型落地和多端覆盖更友好

通信层建议：

- 聊天：先做业务消息能力
- 通话：先做 App 内语音通话
- AI：调用独立 `ai-api-service`

## 十一、接口协作建议

### 与主业务系统

建议用于：

- 登录鉴权
- 用户资料
- 绑定关系
- 风险通知
- 求助记录

### 与 `ai-api-service`

建议用于：

- `/api/fraud-detect`
- `/api/fraud-detect/batch`
- `/api/fraud-detect/chat-log`
- `/api/fraud-detect/link`
- `/api/review`

## 十二、MVP 范围建议

建议第一阶段只做下面这些：

- 双角色登录
- 联系人与绑定关系同步
- 文本聊天
- 可疑消息识别
- 风险提醒
- 一键求助
- 子女端通知与查看

这套范围最适合在短时间内形成演示闭环。

## 十三、后续扩展方向

- 图片 OCR 后识别
- 通话转写后分析
- 风险趋势统计
- 社区协同入口
- 更完善的语音播报与适老交互

## 十四、结论

`silver-shield-mobile-apps` 适合定位为“守护桑榆”体系下的独立移动端项目。

它的价值不在于做一个通用聊天工具，而在于把：

- 老人与子女的通信场景
- AI 诈骗识别能力
- 风险告警与求助联动

真正合并到一个移动端产品里。

如果以落地优先，建议先做 `uni-app` MVP；如果以完整产品为目标，再逐步扩展到通话摘要、OCR 和更多风控能力。
